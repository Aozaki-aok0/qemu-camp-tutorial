## 专业阶段

因为四个方向都不了解，所以就挑了第一个CPU方向。
CPU 方向的核心任务是根据 G233 CPU 指令扩展手册中的指令规格，在 QEMU TCG 为 Xg233ai 扩展实现指令翻译，一共10条。

### QEMU TCG介绍
QEMU支持多种accel（accelerator，加速器，决定 vCPU 的执行方式），但大体可以分为两种：指令模拟技术（TCG）、虚拟化技术（KVM、HVF）等。当使用 TCG 时，QEMU 负责将客户机代码动态翻译为宿主机可执行代码。它会在首次遇到一段客户机代码时，将其转换为宿主机指令集上的可执行代码，并在之后重复利用翻译结果。

### TCG IR介绍
TCG IR相当于QEMU在内部使用的一种“中间语言”。
>Guest 指令
>     ↓
>翻译前端：分析这条指令的语义
>     ↓
>TCG IR
>     ↓
>翻译后端：把 IR 生成为当前 Host 可执行的代码
>     ↓
>Host 机器码

这样一来，QEMU 就不必为每一种 Guest/Host 组合分别写一整套专门的翻译器，而是可以把问题拆成两部分：先把 Guest 指令翻译成 IR，再把 IR 翻译成 Host 代码。

### 指令实现
__QEMU 执行一条指令的简化流程__
>1. 取指 → 2. 译码 → 3. 翻译 → 4. 执行
* __取指__：从模拟内存中读出 32 位机器码。
* __译码__：根据机器码中的字段，识别出它是哪条指令，并提取操作数（寄存器编号、立即数等）。
* __翻译__：把这条指令转换成等价的一个或多个宿主机操作（或调用一个 C 函数）。
* __执行__：实际运行这些操作（或调用 C 函数）。

#### 下面以实验一 test-insn-dma为例
##### 第一步：修改 `target/riscv/insn32.decode`
__代码：__
>`dma      0000110 ..... ..... 110 ..... 1111011 @r`

__作用：教会译码器如何识别这条指令。__
* __位置__：属于流程中的 __第 2 步（译码）__。
* __具体机制__：
QEMU 有一个专门的译码器（由 `decode-insn32.c.inc` 实现，它根据 `insn32.decode` 文件自动生成）。
这一行定义告诉译码器：当你看到一个机器码，它的 `opcode=1111011`、`funct3=110`、`funct7=0000110` 时，就把它识别为指令 `dma`。
同时，它使用模板 `@r` 告诉译码器：这条指令有两个源寄存器（`rs1`, `rs2`）和一个目标寄存器（`rd`）。译码器会自动从机器码的对应位段中提取出这三个寄存器的编号。

##### 第二步：添加翻译函数`trans_dma`（在 `trans_rvi.c.inc` 中）
__代码：__
>`static bool trans_dma(DisasContext *ctx, arg_dma *a)`
>`{`
>    `TCGv dst = get_gpr(ctx, a->rd, EXT_NONE);`
>    `TCGv src = get_gpr(ctx, a->rs1, EXT_NONE);`
>    `TCGv grain = get_gpr(ctx, a->rs2, EXT_NONE);`
>
>    `gen_helper_dma(tcg_env, dst, src, grain);`
>    `return true;`
>`}`

__作用：定义指令的“翻译”动作。__
* __位置__：属于流程中的 __第 3 步（翻译）。__
* __具体机制__：
译码成功后，QEMU 会调用对应指令的 `trans_xxx` 函数。`arg_dma *a` 结构体里就装着译码阶段提取出来的寄存器编号（`a->rd`, `a->rs1`, `a->rs2`）。
这个函数的作用是 生成 __TCG（Tiny Code Generator）中间代码。__
__什么是 TCG？__ 它是一种与宿主机（你的 x86 电脑）无关的中间表示。QEMU 先把 RISC-V 指令翻译成 TCG 操作，然后再把 TCG 操作编译成 x86 机器码去执行。
`get_gpr`：生成 TCG 代码，去读取 RISC-V 通用寄存器的当前值。
`gen_helper_dma`：生成一个 TCG 调用指令，它的目标是一个叫做 `helper_dma` 的 C 函数。

##### 第三步：声明 Helper 函数（在 `helper.h` 中）
__代码：__
>`DEF_HELPER_4(dma, void, env, tl, tl, tl)`

__作用：为第二步的 `gen_helper_xxx` 调用提供“函数原型”，并建立 TCG 世界和 C 语言世界的“桥梁”。__
* __位置__：属于 __第 3 步（翻译）和第 4 步（执行）之间的接口定义__。
* __具体机制__：
    - `DEF_HELPER_4` 是一个宏，它做了两件事：
    1. 声明了 C 函数
    `void helper_dma(CPURISCVState *env, target_ulong arg1, target_ulong arg2, target_ulong arg3)`
    2. 定义了 `gen_helper_dma` 这个宏，并规定了它生成的 TCG 代码应该如何把 TCG 变量（`dst`, `src`, `grain`）映射到 C 函数的参数（`arg1`, `arg2`, `arg3`）。
    - 参数解释：
    `4`：helper 函数总共有 4 个参数。
    `void`：helper 函数没有返回值。
    `env`：第一个参数，类型是 `CPUArchState *`，它是一个指向当前 CPU 所有状态（寄存器、特权级等）的指针。__这是 helper 能够访问模拟器内存、寄存器的关键。__
    `tl`：表示 `target_ulong`，即目标架构（RISC-V 64位）的指针/字长类型。三个 `tl` 正好对应我们传入的三个地址/数值参数。

##### 第四步：实现 Helper 函数（在 `op_helper.c` 中）
__代码:包含转置算法的 C 语言代码。__
__作用：执行指令的真正功能（业务逻辑）。__
* __位置__：属于流程中的 __第 4 步（执行）__。
* __具体机制__：
    - 当 TCG 代码执行到调用点，它会跳转到这个 C 函数中来。
    - 这个函数运行在 QEMU 的进程上下文里，拥有完全的 C 语言能力。
    - 它通过 `env` 指针，调用 QEMU 提供的 API（如 `cpu_ldl_data`）来安全地读写模拟出来的 RISC-V 内存。它不能直接 `*(float*)addr`，因为 `addr` 是模拟内存的地址，不是 QEMU 进程自己的地址。
    - 它完成了矩阵转置的业务逻辑。

#### 后续实验（简洁版）
__2. sort (funct7=22, `0010110`)__
__功能:__ 对 `int32_t` 数组的前 `k` 个元素进行冒泡排序。
__参数:__ `rd`=k（排序长度），`rs1`=数组基址，`rs2`=总长度（未使用）。
__实现：__ `helper` 中读入元素，冒泡排序，写回前 k 个元素。

__3. crush (funct7=38, `0100110`)__
__功能：__ 将 `uint8_t` 数组的低 4 位两两打包为一个字节（先低后高），奇数个时最后一个独立。
__参数：__ `rd`=目标地址，`rs1`=源地址，`rs2`=字节数 n。
__实现：__ 使用 `cpu_ldub_data` / `cpu_stb_data` 逐字节处理。

__4. expand (funct7=54, `0110110`)__
__功能：__ 将每个 `uint8_t` 的低 4 位和高 4 位拆分成两个独立的字节。
__参数：__ `rd`=目标地址，`rs1`=源地址，`rs2`=源字节数 n。
__实现：__ `cpu_ldub_data` 读取，拆分后用 `cpu_stb_data` 写入。

__5. vdot (funct7=70, `1000110`)__
__功能：__ 两个 `int32_t` 向量点积，结果返回 64 位有符号整数。
__参数：__ `rd`=结果（通过返回值），`rs1`=向量 A 基址，`rs2`=向量 B 基址。
__实现：__ `helper` 返回 `target_ulong`，翻译函数中用 `gen_helper_vdot` 生成调用，并用 `gen_set_gpr` 写回 `rd`。

__6. vrelu (funct7=86, `1010110`)__
__功能：__ 对 `int32_t` 向量逐元素做 `ReLU（max(x, 0)）`。
__参数：__ `rd`=目标地址，`rs1`=源地址，`rs2`=长度 n。
__实现：__ `helper` 循环比较并写回，支持 in‑place。

__7. vscale (funct7=102, `1100110`)__
__功能：__ `int32_t` 向量乘以一个标量。
__参数：__ `rd`=目标地址，`rs1`=源地址，`rs2`=标量乘数。
__实现：__ `helper` 中逐元素执行 `(int64_t)a[i] * scale `并截断写回。

__8. vmax (funct7=118, `1110110`)__
__功能：__ 返回 `int32_t` 向量中的最大值（64 位符号扩展）。
__参数：__ `rd`=结果（返回值），`rs1`=源地址，`rs2`=元素个数 n。
__实现：__ `helper`返回 `target_ulong`，翻译函数中用 `tcg_temp_new` 接收返回值并写回 `rd`。

__9. gemm (funct7=14, 0001110)__
__功能：__ 4×4 `int32_t`矩阵乘法 C = A × B。
__参数：__ `rd`=C 基址，`rs1`=A 基址，`rs2`=B 基址。
__实现：__ `helper` 中三层循环乘加，使用 `int64_t` 累加防止溢出。

__10. vadd (funct7=30, `0011110`)__
__功能：__ 两个 `int32_t` 向量逐元素加法。
__参数：__ `rd`=目标地址，`rs1`=A 基址，`rs2`=B 基址。
__实现：__ `helper` 循环相加写回，支持 in‑place。

## 总结

* 掌握了 QEMU TCG 自定义指令 添加的标准流程（decode → trans → helper）。

* 理解了 insn32.decode 模板的作用、翻译函数与 TCG 变量、helper 的声明与调用机制。

* 能使用 QEMU 内存访问函数 (cpu_ld*/cpu_st*) 实现矩阵转置、排序、向量运算等。

* 学会了通过 gen_helper_xxx 生成 helper 调用、以及如何处理有返回值的 helper（如 vdot、vmax）。
